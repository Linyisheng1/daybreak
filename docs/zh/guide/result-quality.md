# 成果质量

Daybreak 的智能体、方法论技能和沙箱工具用于辅助安全人员发现线索、复现问题和整理证据。自动化输出不是最终漏洞结论，尤其不能仅凭版本、响应头、错误页面或扫描器模板命中确认高危漏洞。

## 发现状态

| 状态 | 含义 | 是否进入正式风险统计 |
| --- | --- | --- |
| `suspected` | 存在线索，但证据或影响尚未完整验证 | 否 |
| `validated` | 已稳定复现并确认实际安全影响 | 是 |
| `false_positive` | 已复核并排除 | 否 |

当前版本尚未在服务端强制执行全部证据规则。操作者应在报告交付前复核状态和严重性。

## 最低确认标准

确认一个漏洞至少需要：

1. 明确的受影响资产、入口、参数和身份条件。
2. 可由其他测试人员重复执行的步骤。
3. 请求/响应、命令输出、截图或文件等原始证据。
4. 实际影响证明，而不是只描述理论风险。
5. 对缓存、权限差异、网络异常和工具误报等常见原因的排除。

高危和严重漏洞还应进行第二次复现；条件允许时，由不同操作者或不同智能体独立复核。

## 严重性基线

| 级别 | 参考条件 |
| --- | --- |
| 信息/低危 | 信息暴露或配置弱点，未证明直接安全影响 |
| 中危 | 已复现，对单个用户、有限数据或局部功能造成影响 |
| 高危 | 已证明越权、敏感数据访问、账户接管、文件读写、内网访问等明显影响 |
| 严重 | 已证明可低门槛造成大范围接管、远程代码执行、核心数据泄露或关键攻击链推进 |

仅有扫描器命中、组件版本或公开 CVE 匹配时，应保留为疑似，并在完成针对目标的实际验证后再调整等级。

## 报告前检查

- 高危和严重发现是否均为 `validated`。
- 每个已确认发现是否具备原始证据和实际影响。
- 重复发现是否已合并，证据是否追加到同一记录。
- 疑似项是否与正式漏洞分开展示。
- 工具执行是否产生有效输出，而不只是成功退出码。
- 敏感凭据、个人信息和客户数据是否已脱敏。

## 沙箱能力

方法论技能描述测试流程，沙箱镜像提供实际执行能力。技能存在不代表镜像中一定安装了同名工具。执行前应确认所需命令或浏览器能力可用；工具缺失、空输出或运行异常时，应记录阻塞原因，不应据此确认漏洞。
